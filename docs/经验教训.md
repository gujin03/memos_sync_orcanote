# Memos → Orca Note 同步插件开发经验与教训

> 本文档记录开发过程中遇到的关键问题及其解决方案，供后续参考。

---

## 一、4层结构设计（方案B）

### 1.1 最终确定的同步结构

每条 Memos 同步到虎鲸笔记后，采用 4 层独立结构：

```
第1层（根块）：{Memos_ID}         ← 仅 #Memos Note 标签，ID 存标签属性
├── 第2层（标题块）：{标题/摘要}    ← 核心标签（#zk, #career 等）
│    └── 第3层（内容块）：正文段落  ← 按逻辑分段，标题块的子块
└── 第4层（链接块）：🔗 Open in Memos  ← 链接，最后
```

### 1.2 设计决策

| 决策 | 选项 | 选择理由 |
|------|------|---------|
| 链接位置 | 右1 × 第4层 √ | 根块保持干净(机器可读)，链接不干扰阅读流 |
| 标签在哪 | 根块 × 标题块 √ | 知识检索锚点是标题，不是 ID |
| ID 存储 | 文本中 × 标签属性 √ | `query({name:"Memos Note", properties:[{name:"ID", op:1, value:"xxx"}]})` 精确匹配 |
| `←` 箭头 | 保留 × 移除 √ | 层级由子块关系天然表示，不需要显式标记 |
| 内容格式 | HTML × 纯文本 √ | 避免 `<p>` 等标签的渲染问题 |
| 链接格式 | 显示URL × 隐藏URL √ | 用 `{t:"l", v:"🔗 Open in Memos", l:url}` 内联链接，仅显示文字 |

## 二、Orca 核心 API 踩坑记录

### 2.1 `insertBlock` 的参数陷阱

**坑 1：`#` 符号导致 "(intermediate value) is not iterable"**

```typescript
// ❌ 内容文本中包含 #Memos Note 会被 Orca 当作标签引用解析
const rootId = await invokeEditorCommand("core.editor.insertBlock", null, inbox, "lastChild",
  [{ t: "t", v: `${memoId} #Memos Note` }],
)  // → "is not iterable"

// ✅ 只传纯文本，标签通过 insertTag 独立添加
const rootId = await invokeEditorCommand("core.editor.insertBlock", null, inbox, "lastChild",
  [{ t: "t", v: memoId }],
)  // → 正常
await invokeEditorCommand("core.editor.insertTag", null, rootId, noteTag, [{ name: "ID", type: 1, value: memoId }])
```

**根因**：Orca 的 `blockConvert` 在将 ContentFragment 转换为纯文本时，发现 `#` 开头的文本会尝试解析为标签引用。如果标签解析过程中遇到非 iterable 的值，就抛出 `(intermediate value) is not iterable`。

**教训**：ContentFragment 的文本内容中不要包含 `#` 符号。标签必须通过 `insertTag` 独立添加，永远不要写在文本里。

**坑 2：`{ type: "text" }` repr 导致异常**

```typescript
// ❌ 显式传 text repr
invokeEditorCommand("core.editor.insertBlock", null, parentBlock, "lastChild",
  [{ t: "t", v: title }],
  { type: "text" },  // ← 异常
)

// ✅ 不传 repr（默认 text 类型）
invokeEditorCommand("core.editor.insertBlock", null, parentBlock, "lastChild",
  [{ t: "t", v: title }],
)
```

**教训**：Orca 中 text 是默认类型，不需要显式指定 `{type:"text"}`。只有非 text 类型（image、code 等）才需要传 repr。

### 2.2 `insertBlock` 的引用脏数据问题

**问题**：标题块和链接块都使用 `lastChild` 插入，但链接块跑到了标题块前面。

**根因**：`orca.state.blocks[rootId]` 返回的 Block 引用在插入标题块后变为**脏引用**——`children` 数组未更新。插入链接时，`insertBlock` 的 `lastChild` 逻辑读到空 `children`，导致位置错乱。

```typescript
// ❌ 用旧引用插入链接块
const titleId = await insertBlock(..., noteBlock, "lastChild", ...)
// noteBlock.children 仍为空（引用未刷新）
await insertBlock(..., noteBlock, "lastChild", ...)  // → 链接跑到了标题前面

// ✅ 刷新引用后再插入
noteBlock = orca.state.blocks[rootId] ?? noteBlock  // 重新获取
await insertBlock(..., noteBlock, "lastChild", ...)  // → 链接在最后
```

**教训**：`orca.state.blocks[id]` 的引用**不会自动更新**。多次对同一父块插入子块时，每次插入前都要重新获取父块引用。

### 2.3 `invokeGroup` 内部的兼容性问题

**问题**：在 `invokeGroup` 内部调用 `insertBlock` 抛出 `(intermediate value) is not iterable`，移除 invokeGroup 后正常。

**根因**：`invokeGroup` 为了撤销功能做了批量事务处理，可能与 `insertBlock` 内部的 `blockConvert` 或 `CreateBlock` 后端操作产生冲突。

**教训**：如果同步操作只需要一次性写入、不需要撤销支持，就不要用 `invokeGroup` 包裹。

### 2.4 `deleteBlocks` + 重建的坑

**问题**：想通过"删除旧块→创建新块"的方式来升级结构，但删除操作的错误被 try-catch 吞掉，导致后续 insertBlock 操作在新的空位置上生效。

**根因**：涉及到块结构变更（删除 → 重建）时，`deleteBlocks` 和后续的 `insertBlock` 之间可能存在状态不一致。

**教训**：对于旧结构不兼容的升级场景，要么用 MCP 批量删除后全量同步，要么在插件代码中确保删除和重建在同一个事务上下文中进行。

## 三、链接块渲染

### 3.1 链接 ContentFragment

**需求**：第4层链接块显示 `🔗 Open in Memos`，点击跳转到 Memos 原文。

**方案**：使用 Orca 的内联链接 ContentFragment：

```typescript
// t: "l" = 内联链接，v = 显示文本，l = 跳转 URL
{ t: "l", v: "🔗 Open in Memos", l: "https://memos.19491949.xyz/m/xxx" }
```

**原理**：Orca 的 HTML 转换器将 `t: "l"` 渲染为 `<a href="${Jn.l}">${convertFormats(Jn)}</a>`。

**对比**：
| 方式 | 效果 | 推荐 |
|------|------|------|
| `{t:"t", v:"🔗 Open in Memos：URL"}` | 显示完整 URL，文本 | ❌ |
| `{t:"t", v:"[text](url)"}` | 不解析Markdown，显示原文 | ❌ |
| `{t:"l", v:"🔗 Open in Memos", l:url}` | 渲染为可点击链接，只显示文字 | ✅ |

## 四、查重机制

### 4.1 标签属性查询

通过 `#Memos Note` 标签的 `ID` 属性精确查找已有块：

```typescript
const resultIds = await orca.invokeBackend("query", {
  q: { kind: 1, conditions: [
    { kind: 4, name: "Memos Note", properties: [{ name: "ID", op: 1, value: memoId }] },
  ]},
  pageSize: 1,
})
```

**优势**：精确匹配，不会误伤其他文本中包含 ID 的块。

### 4.2 全量同步的清理策略

- 旧格式不兼容（html 内容 + 标签格式不同），无法原地升级
- 策略：**MCP 批量删除所有 `#Memos Note` 块 → 全量同步重建**
- 插件内不处理旧块升级，统一按新结构创建

## 五、部署注意事项

### 5.1 缓存与双路径部署

**关键问题**：Orca 加载 `dist/index.js`（旧版），而非根目录的 `index.js`（新版）。

**原因**：插件解压后入口文件在 `dist/` 下。`npm run build` 的输出到 `dist/index.js`，但复制时必须同时更新两个位置。

```bash
npm run build
cp dist/index.js /path/to/plugins/{plugin}/index.js        # 根目录
cp dist/index.js /path/to/plugins/{plugin}/dist/index.js    # dist 目录（Orca 实际加载）
```

**验证方式**：对比 MD5 确保两个文件一致。

### 5.2 浏览器缓存

Orca Note 加载插件时使用 `?_t=时间戳` 缓存策略。如果时间戳不变，可能加载旧版本。强制重载插件可清除缓存。

## 六、调试技巧

### 6.1 Orca MCP 查询

使用 Orca MCP 可以实时检查块结构：

```bash
# 查询块及其子块
mcp__orca-note__get_blocks_text(blockIds=[xxx])

# 查询带特定标签的块
mcp__orca-note__query_blocks(q: {kind:100, conditions: [{kind:4, name:"Memos Note"}]})
```

### 6.2 批量清理旧数据

```bash
# 查所有 #Memos Note 块
query({kind:100, conditions: [{kind:4, name:"Memos Note"}]})

# 分批删除
delete_blocks(blockIds=[...])
```

### 6.3 错误定位

- `(intermediate value) is not iterable` → 通常是 `insertBlock` 的 content 或 repr 参数有 `#` 符号
- `No converter registered for plain-undefined` → repr 参数位置错位
- 报错在 `blockConvert` → 检查 ContentFragment 的格式

## 七、事务性 vs 顺序性

| 场景 | 推荐策略 |
|------|---------|
| 批量插入块（需保序） | 逐条 await，不用 invokeGroup |
| 批量删除旧块 | 可一次性 deleteBlocks，删除后重建 |
| 精确查重 | 用 tag property query，不用文本匹配 |
| 块引用刷新 | 每次 insertBlock 前重新从 state 获取 |
